# Worklog Typora 风格一体化编辑器设计

## 背景

当前编辑区由原生 `textarea` 和 `ReactMarkdown` 预览分支组成，用户需要通过右上角按钮在“编辑”和“预览”之间切换。这个结构有两个直接问题：

- 编辑和阅读是分离的，写作时需要频繁切换心智模型。
- 原生 `textarea` 的排版和光标表现完全受宿主 WebView 影响，已经出现过多行输入时的光标异常。

这次目标不是继续修补原生输入框，而是直接换成一个成熟、简单、开源、接近 Typora 的 Markdown 编辑器，让“编辑”和“预览”合成一个区域。

## 用户目标

- 编辑区要尽量像 Typora：边写边渲染，而不是“左边写源码、右边看结果”。
- 交互要极简：不要现有的“编辑 / 预览”切换按钮。
- 保留现有 Worklog 的核心行为：
  - 仍然保存为 Markdown 字符串
  - 仍然按日期切换文档
  - 仍然使用当前自动保存和手动 flush 逻辑

## 非目标

- 本次不引入评论、协作、附件上传、图片管理、表格工具栏等扩展能力。
- 本次不改动 Worklog 的文件格式、目录结构和 Tauri 后端接口。
- 本次不重做整个主界面布局，只替换主编辑区。
- 本次不追求完全复刻 Typora 的全部交互，只追求“接近 Typora 的一体化写作体验”。

## 候选方案

### 方案 A：Vditor `ir` 模式

优点：

- 原生支持 `ir`（即时渲染）模式，目标交互和 Typora 最接近。
- 仍以 Markdown 为底层内容格式，和当前持久化模型天然兼容。
- 可隐藏工具栏、计数器、缓存等默认 UI，能收成极简样式。
- 提供实例 API 和输入回调，适合包成一个轻量 React 包装组件。

代价：

- 编辑器本体能力较多，需要额外覆盖一层默认样式，避免界面显得太“编辑器”。
- 需要用 imperative API 管理实例，不能像原生 `textarea` 那样完全受控。

### 方案 B：Milkdown / Crepe

优点：

- WYSIWYG 能力强，也能做接近 Typora 的体验。
- 现代化程度高，可扩展性强。

代价：

- 更像编辑器框架，集成和定制成本明显高于当前项目所需。
- 对这个小型本地日记应用来说偏重，不符合“简单很重要”的约束。

### 方案 C：`@uiw/react-md-editor`

优点：

- React 接入很轻。
- 能较快替换掉当前 `textarea + preview` 结构。

代价：

- 更接近“Markdown 编辑器 + 预览能力”，而不是 Typora 风格的一体化即时渲染。
- 无法真正满足这次对“编辑和预览合一”的核心诉求。

## 决策

采用 **方案 A：Vditor `ir` 模式**。

核心原因只有两个：

1. 它最贴近这次想要的交互结果。
2. 它不要求我们重写当前的内容状态和保存链路。

## 总体设计

### 一、编辑器边界

新增一个专门的包装组件：

- 新建：`src/components/editor/vditor-editor.tsx`

这个组件只负责两件事：

- 把 Vditor 实例挂到 React 生命周期里
- 把编辑器内容桥接成当前应用仍在使用的 `string` 值

建议 props 保持极小：

- `value: string`
- `onChange: (nextValue: string) => void`
- `onBlur?: () => void`

不把自动保存、日期切换、状态栏逻辑塞进包装组件里，避免第三方编辑器和业务流程耦合。

### 二、App 里的改动边界

`src/App.tsx` 只做以下收敛：

- 删除 `EditorMode` 状态
- 删除 `mode === "edit" ? textarea : article` 分支
- 删除右上角“编辑 / 预览”切换按钮
- 删除 `Cmd/Ctrl+P` 预览切换快捷键
- 把当前编辑区替换为 `<VditorEditor value={content} onChange={handleContentChange} />`

保留以下逻辑不动：

- `content` 仍然是主状态
- `handleContentChange` 仍然是所有编辑输入进入应用状态的唯一入口
- `flushEntry`、`loadEntry`、自动保存 timer、日期切换前 flush、窗口关闭前 flush 全部继续由 `App.tsx` 管

也就是说，编辑器换了，但 Worklog 仍然把它当成“会吐出 Markdown 字符串的输入组件”。

### 三、Vditor 包装组件内部设计

包装组件采用“实例驱动 + 受控同步”的桥接模型。

内部需要的最小状态/引用：

- `hostRef`：挂载编辑器 DOM
- `instanceRef`：保存 Vditor 实例
- `lastAppliedValueRef`：记录最近一次由 React 主动写回编辑器的值
- `isSyncingRef`：防止 `setValue` 触发回流造成重复 `onChange`

行为规则：

1. 组件首次挂载时创建 Vditor 实例，固定使用 `mode: "ir"`。
2. 编辑器初始化完成后，用外部 `value` 作为初始内容。
3. 用户输入时，编辑器通过回调把最新 Markdown 字符串传给 `onChange`。
4. 外部 `value` 变化时，仅当它和当前实例内容不一致时才执行 `setValue`。
5. 组件卸载时销毁实例，防止切日期或测试环境残留 DOM/事件。

这样可以保证：

- React 仍然是业务真源
- 编辑器不会因为每次渲染都被重建
- 外部加载新日期内容时，编辑器能被可靠同步

## 交互设计

### 一、视觉目标

编辑区要接近“文稿页面”，而不是“IDE 风格编辑器”。

因此 Vditor 的 UI 需要做减法：

- 隐藏工具栏
- 隐藏内置计数器和状态条
- 关闭默认缓存
- 去掉厚边框、分栏、阴影和明显的编辑器面板感

主页面最终结构保持：

- 日期标题
- 日期切换控件
- 分隔线
- 错误提示
- 一体化编辑区
- 右下角保存状态点

编辑区内部不再有“编辑 / 预览”二级控制。

### 二、快捷键

保留：

- `Cmd/Ctrl+S`：立即 flush 当前内容
- 现有设置快捷键和侧栏快捷键

移除：

- `Cmd/Ctrl+P` 的预览切换行为

原因很简单：切换目标已经不存在，再保留这个快捷键只会制造困惑。

### 三、空状态

当前应用切到一个未创建日期时，`content` 仍然会由现有读档逻辑提供默认内容或空内容。新的编辑器不需要单独实现“空日记创建”流程，只要正确显示 `value` 即可。

是否写盘、何时创建文件，仍由现有 `saveEntry` 路径决定。

## 样式设计

### 一、样式文件边界

- `src/components/editor/vditor-editor.tsx`
  - 负责引入 Vditor 官方 CSS
- `src/styles.css`
  - 负责覆盖 Worklog 所需的极简主题样式

### 二、需要清理的现有样式

这几个现有块会被删除或失效：

- `.editor-stage textarea`
- `.markdown-preview`
- `.editor-actions`
- `.mode-switch`

### 三、新的样式方向

新的 `.editor-stage` 仍然负责页面级留白和高度控制，但内部宿主改成 Vditor 容器。

需要覆盖的重点：

- 容器背景改为透明或接近页面底色
- 边框改为无感或极轻
- 工具栏完全隐藏
- `ir` 编辑区正文的字号、行高、段落间距对齐当前阅读区风格
- 代码块、列表、标题的默认排版保留 Markdown 文稿感，而不是后台管理系统风格

目标不是造一套新设计系统，而是让第三方编辑器看起来像本来就长在 Worklog 里。

## 对现有依赖和代码的影响

### 删除或减少使用

在主编辑区替换完成后，`src/App.tsx` 中以下内容应被移除：

- `ReactMarkdown`
- `remark-gfm`
- `Pencil`
- `Eye`
- `mode` 相关状态和 JSX

注意：如果项目其他位置仍然使用这些依赖，则不应误删；但按当前代码结构，主窗口移除后它们大概率可以一并清理。

### 新增依赖

新增 Vditor 运行时依赖，并只在主窗口编辑区使用。

这次不额外再引入 React 封装库，避免多包一层。直接在本项目里写一个很薄的包装组件，控制面更小，也更符合“简单优先”。

## 测试设计

第三方编辑器接入的最大风险不是功能本身，而是把现有自动保存链路弄断。因此测试要分成两层。

### 一、包装组件测试

新增：

- `src/components/editor/vditor-editor.test.tsx`

测试重点：

- 挂载时会创建编辑器实例
- 传入 `value` 变化时会同步到实例
- 实例触发输入回调时会把 Markdown 字符串传给 `onChange`
- 卸载时会销毁实例

这里不跑真实 Vditor DOM 行为，而是 mock `vditor` 模块，测试我们自己的桥接逻辑。

### 二、App 集成测试

修改：

- `src/App.test.tsx`

测试重点：

- 主界面不再渲染“编辑 / 预览”切换按钮
- 主编辑区改为一体化编辑器宿主，而不是 `textarea`
- 通过编辑器回调更新内容后，现有 `600ms` 自动保存仍然生效
- 切日期前 flush、空内容处理、失败重试等已有自动保存语义仍然成立

集成测试里同样 mock `vditor`，不要依赖真实编辑器在 jsdom 中运行。

## 风险与缓解

### 风险 1：实例同步造成内容回环

场景：

- React 外部 `value` 变化后调用 `setValue`
- 编辑器又把这次程序写入当成用户输入抛回 `onChange`
- 最终引发重复保存或脏状态抖动

缓解：

- 在包装组件里显式区分“程序同步”和“用户输入”
- 只有真实用户输入才向上触发 `onChange`

### 风险 2：Vditor 默认样式过重，破坏当前界面气质

缓解：

- 严格隐藏工具栏和状态 UI
- 只保留正文区域
- 用现有 Worklog 的字号、行高和留白覆盖正文排版

### 风险 3：真实编辑器难以在测试环境运行

缓解：

- 单元测试和集成测试都 mock `vditor`
- 测试重点放在“桥接逻辑”和“保存链路”而不是真实编辑器渲染细节

## 实施顺序

1. 新增 Vditor 包装组件和对应测试，先把桥接边界稳定下来。
2. 在 `App.tsx` 中用包装组件替换当前 `textarea + preview` 分支。
3. 删除模式状态、切换按钮和对应快捷键。
4. 清理 `styles.css` 中旧编辑区样式，补上 Vditor 极简覆盖。
5. 跑编辑器包装测试和现有 `App` 自动保存测试，确认保存链路没有回归。

## 验收标准

- 主编辑区不再出现“编辑 / 预览”切换按钮。
- 用户进入编辑区后直接得到一体化写作体验，不需要切模式。
- 日期切换、自动保存、失败提示、快捷键保存都维持当前行为。
- 编辑器视觉风格足够克制，整体感受接近 Typora，而不是工具型 Markdown IDE。
