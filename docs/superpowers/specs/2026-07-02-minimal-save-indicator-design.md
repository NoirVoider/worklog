# Worklog 极简自动保存指示器设计

## 背景

`2026-07-02-sidebar-autosave-design.md` 已经定义了自动保存、flush、失败保留内容等核心行为，但当时的状态展示仍然延续了“底部状态栏 + 文案”的思路。

这会带来两个体验问题：

- 文案会持续占据注意力，即使编辑器当前没有需要用户处理的事情。
- 成功态和待保存态都通过文字表达，视觉重量偏高，不符合更专注的写作界面目标。

本次方案只收敛自动保存状态的可视反馈，不改动自动保存时机、文件读写链路、失败处理和切日期保护。

## 与现有 autosave 方案的关系

- 本文档覆盖 `2026-07-02-sidebar-autosave-design.md` 中“状态展示”一节的展示决策。
- 自动保存延迟仍为约 `600ms`。
- 切日期前 flush、失败不切走、旧请求保护等行为维持不变。

## 目标

- 把底部状态反馈改成极简、低干扰的 icon-only 方案。
- 去掉中文状态文案，内部状态统一改为英文语义标识。
- 保存成功后不再闪出成功文案，尽快回到无提示状态。
- 只有“系统正在忙”或“确实失败了”时，才占用用户的注意力。

## 非目标

- 本次不改自动保存的时机与 debounce 参数。
- 本次不增加 toast、通知中心、悬浮提示等新反馈层。
- 本次不引入用户可配置的状态样式。
- 本次不改底部状态容器的位置，只改其内容与语义。

## 决策摘要

采用以下方案：

1. 内部状态枚举改成英文语义值。
2. 底部状态区只显示小 icon，不再显示任何可见文字。
3. `loading`、`creating`、`pending`、`saving` 共用同一类微型 spinner。
4. `failed` 显示一个微型 warning icon。
5. `idle`、`empty`、`saved` 不显示任何 icon。
6. 保存成功后直接回到安静态，不再额外展示 `Saved` 成功反馈。

## 状态模型

### 新的状态标识

- `idle`
- `loading`
- `empty`
- `pending`
- `saving`
- `saved`
- `failed`
- `creating`

### 旧状态到新状态的映射

- `就绪` -> `idle`
- `读取中` -> `loading`
- `未创建` -> `empty`
- `未保存更改` -> `pending`
- `自动保存中` -> `saving`
- `已自动保存` -> `saved`
- `自动保存失败` -> `failed`
- `创建中` -> `creating`

### 推荐运行时语义

- 初次读取某一天时进入 `loading`。
- 读取完成且文件存在时进入 `idle`。
- 读取完成但文件不存在时进入 `empty`。
- 用户一旦输入并产生 dirty 内容，立即进入 `pending`。
- debounce 触发并发出保存请求时进入 `saving`。
- 保存成功后直接回到 `idle`。
- 保存失败时进入 `failed`，直到用户再次编辑或下一次保存成功。
- 主动创建文件时进入 `creating`，完成后回到 `idle` 或 `empty` 对应的稳定态。

说明：

- `saved` 保留为语义映射值，便于从旧文案状态迁移到英文状态。
- 运行时在保存成功后必须直接落到 `idle`，不在 UI 层停留 `saved`，以避免状态闪烁。

## 展示规则

### 可见性

只有以下两类情况允许出现可见反馈：

- 系统正在后台处理：`loading`、`creating`、`pending`、`saving`
- 用户需要注意失败：`failed`

以下状态必须保持完全安静：

- `idle`
- `empty`
- `saved`

### 图标映射

- `loading` -> spinner
- `creating` -> spinner
- `pending` -> spinner
- `saving` -> spinner
- `failed` -> warning icon
- `idle` -> hidden
- `empty` -> hidden
- `saved` -> hidden

### 视觉约束

- icon 尺寸控制在 `10px` 到 `12px` 之间。
- spinner 使用低对比度中性色，避免抢占正文层级。
- spinner 旋转节奏应平缓，不做强烈脉冲或放大缩小。
- warning icon 比 spinner 稍高一档对比度，但仍保持克制。
- 状态区始终只容纳一个 icon，不再出现图标 + 文案并列。

## 交互细节

### Pending 与 Saving 的关系

`pending` 和 `saving` 都使用 spinner，是有意为之。

原因：

- 从用户角度，这两种状态都表示“系统正在为写入做准备或正在写入”。
- 区分不同动画或不同图标，会让界面重新变复杂。
- 用户当前最关心的是“是否需要我介入”，而不是精确区分“等 debounce”还是“请求已发出”。

### 成功态策略

保存成功后不再展示成功文案或成功图标。

原因：

- 成功是默认预期，不应持续打断写作视线。
- 成功态一闪而过会产生不必要的动态噪声。
- 用户一旦停止看到 spinner，就足够理解“这次保存已经完成”。

### 失败态策略

失败态需要被看见，但仍尽量轻量：

- 使用单个 warning icon。
- 保留 `title` 和可访问性标签，例如 `Save failed`。
- 失败后不自动弹 toast。
- 下一次编辑发生时，可先回到 `pending`，让系统自动重试链路继续接管。

## 可访问性

- 状态容器仍应保留语义角色，例如 `role="status"`。
- 图标本身需要有英文 `aria-label`，而不是完全静默。
- `failed` 可额外提供 `title="Save failed"`，方便 hover 时查看。
- 视觉上不显示文字，不等于对辅助技术完全移除文本语义。

## 对实现的影响

- `src/App.tsx`
  - `StatusMessage` 从中文文案改为英文语义值。
  - 保存成功路径从“展示成功文案并延迟回落”改为直接回到 `idle`。
  - 底部状态区从文本渲染改为基于状态渲染 icon 或空内容。
- `src/styles.css`
  - 状态区样式改成适配单 icon 的尺寸和对齐。
  - 增加 spinner 与 warning icon 的轻量样式。
- `src/App.test.tsx`
  - 更新状态断言，不再依赖中文文案。
  - 新增 icon-only 的渲染测试。
  - 覆盖 `pending`、`saving`、`failed`、`idle` 的显隐行为。

## 测试计划

### 状态映射

- 断言输入后状态进入 `pending`。
- 断言 debounce 到时后状态进入 `saving`。
- 断言保存成功后不再显示任何可见状态元素。
- 断言保存失败后会显示 warning icon。

### UI 呈现

- 断言底部状态区不再包含可见状态文案。
- 断言同一时刻最多只出现一个 icon。
- 断言 `idle` / `empty` / `saved` 时状态区为空。

### 可访问性

- 断言 spinner 和 warning icon 具有正确的 `aria-label`。
- 断言失败态带有 `title` 或等价辅助语义。

## 验收标准

- 正常编辑时，底部状态反馈的存在感明显低于当前文本状态栏。
- 用户输入后能看到轻量 spinner，理解“系统正在处理保存”。
- 保存成功后状态反馈自然消失，不出现成功文字闪烁。
- 保存失败时用户仍能快速注意到异常，但不会被 toast 或重提示打断。
- 界面上不再出现中文自动保存状态文案。
