# Worklog 侧栏日期列表精简与删除交互设计

## 背景

当前侧栏下半部分的日期卡片列表存在三个问题：

- 顶部固定的“最近”标题信息量很低，占掉了本来就不宽裕的垂直空间。
- 列表目前只渲染 `normalizedEntries.slice(0, 8)`，既不围绕当前关注日期展开，也没有继续加载更多内容的能力。
- 卡片没有删除入口，用户无法直接从侧栏管理某一天的内容。

同时，这个项目的数据模型目前只有稳定的 `date` 维度，没有 `created_at` 或 `updated_at` 字段。为了做“最新创建排序”去引入一套元数据体系，会明显增加复杂度，不符合本轮“保持轻量”的目标。

## 目标

- 去掉侧栏列表上的“最近”标题，让卡片区更直接。
- 列表展示逻辑从“固定前 8 条”改成“围绕当前关注日期展开”的轻量懒加载。
- 为卡片增加右键小菜单，支持删除某一天。
- 删除当前选中日期后，编辑区仍停留在这一天，并进入空状态，方便用户立即重写。

## 非目标

- 本次不引入虚拟列表。
- 本次不引入“按最新创建”或“按最近修改”排序。
- 本次不引入多操作菜单、标签、批量管理等复杂侧栏能力。
- 本次不新增 toast 系统，只复用现有错误提示区域。

## 决策摘要

采用以下方案：

1. 去掉侧栏列表的“最近”标题。
2. 列表继续按日期排序，不改成“最新创建排序”。
3. 列表锚点使用“当前选中日期”，首次进入时默认等于今天。
4. 不做虚拟列表，只做分页懒加载。
5. 初始窗口渲染锚点附近一小段卡片，滚动接近边缘时再向对应方向扩展。
6. 卡片右键弹出自定义小菜单，当前版本只提供一个动作：`删除这一天`。
7. 删除当前选中日期后，不跳转到别的日期，编辑区保留当前日期并进入“这一天还没有日记”的空状态。

## 一、列表模型

### 排序原则

- 列表仍然按日期顺序展示，保持与现有 `listEntries()` 和 `normalizeEntries()` 的时间心智一致。
- 不新增“创建时间”概念。
- 不读取文件系统元数据来伪造“最新创建排序”。

### 锚点规则

- 如果当前 `selectedDate` 在已有条目列表中存在，锚点就是该条目所在位置。
- 如果当前 `selectedDate` 不在列表里：
  - 例如这一天还没创建；
  - 或者刚被删除；
  - 则按日期顺序计算它理论上的插入位置，并把该位置作为锚点。
- 首次进入页面时，由于 `selectedDate` 默认就是今天，因此列表也自然以今天为中心。

### 渲染窗口

- 不再直接渲染 `normalizedEntries.slice(0, 8)`。
- 改为根据锚点索引和当前已展开范围，切出一个连续窗口进行渲染。
- 窗口本身只是一段数组切片，不额外引入复杂缓存层。

## 二、懒加载策略

### 初始窗口

- 初始先渲染 `24` 条。
- 初始窗口尽量以前后各半的方式围绕锚点展开。
- 如果锚点靠近列表边缘，就自动把不足的一侧补到另一侧，保证尽量凑满 `24` 条。

### 后续加载

- 每次扩展 `12` 条。
- 接近顶部时，补更“新”的日期。
- 接近底部时，补更“旧”的日期。
- 触发阈值固定为距离顶部或底部 `120px` 时触发。

### 滚动稳定性

- 向底部追加旧日期时，正常追加即可。
- 向顶部补新日期时，必须保持用户当前视口位置稳定。
- 做法是：
  - 记录补数据前的 `scrollHeight`
  - 扩展窗口
  - 在下一帧用新的 `scrollHeight - oldScrollHeight` 修正 `scrollTop`
- 这样列表不会因为 prepend 而出现“向下蹦一下”的问题。

### 窗口重置时机

- 用户通过日历、前后一天按钮或点击卡片切换日期时，窗口重新围绕新的 `selectedDate` 展开。
- 用户只是在编辑正文时，不重置窗口。
- 删除某一项后，窗口围绕当前 `selectedDate` 重新计算。

## 三、卡片右键删除交互

### 菜单形式

- 不在卡片表面增加常驻删除按钮。
- 用户在卡片上右键时，弹出自定义小菜单。
- 本轮菜单只包含一个动作：
  - `删除这一天`

### 菜单行为

- 菜单默认贴近鼠标位置出现。
- 如果菜单靠近窗口边缘，则自动向内偏移，避免被裁切。
- 以下动作会关闭菜单：
  - 点击空白处
  - 按 `Esc`
  - 滚动列表
  - 选择菜单项

### 删除确认

- 点击 `删除这一天` 后，弹一个很轻的确认层。
- 文案保持直接：
  - `删除 2026-07-02？此操作不可恢复`
- 操作只有两个：
  - `取消`
  - `删除`

### 删除后的结果

- 删除非当前选中卡片：
  - 刷新列表
  - 编辑区不动
- 删除当前选中卡片：
  - 不跳到其他日期
  - 当前 `selectedDate` 保持不变
  - 重新读取该日期时返回 `exists: false`
  - 编辑区直接进入“这一天还没有日记”的空状态

这个策略的好处是，用户误删后如果想当场重写，不会被突然带去别的日期。

## 四、后端删除能力与数据流

### 前端接口

`src/lib/api.ts` 增加：

- `deleteEntry(date: string): Promise<void>`

mock API 中：

- 从 `mockEntries` 里删除对应日期。

Tauri API 中：

- 新增 `invoke<void>("delete_entry", { date })`

### Rust 命令

`src-tauri/src/lib.rs` 增加：

- `#[tauri::command] fn delete_entry(app: AppHandle, date: String) -> Result<(), String>`

实现保持简单：

- 先通过现有 `entry_path()` 解析目标文件。
- 如果文件存在，则删除 `daily/<date>/daily.md`
- 然后尝试删除空目录：
  - `daily/<date>/`
- 如果目录非空或不存在，不把它视为错误。

不做额外目录扫描或深层清理，保持逻辑最小。

### 前端删除流程

前端点击确认删除后：

1. 关闭确认层和右键菜单
2. 调用 `worklogApi.deleteEntry(date)`
3. `await refreshEntries()`
4. 如果删的是当前选中日期：
   - 调用现有 `loadEntry(selectedDate)`
5. 如果删的不是当前选中日期：
   - 只刷新列表窗口和滚动范围

## 五、失败处理

失败处理保持简单，不额外加 toast：

- 删除失败时：
  - 保留当前编辑内容与选中状态
  - 关闭菜单与确认层
  - 通过现有 `error-ribbon` 显示错误信息
- 列表刷新失败时：
  - 不清空当前已显示卡片
  - 保留已有窗口内容
  - 同样通过 `error-ribbon` 反馈

这样用户即使删除失败，也不会突然丢上下文。

## 六、测试边界

测试保持轻量，只覆盖本轮新增行为：

### 组件行为

- 不再渲染“最近”标题。
- 初始列表不是固定 `8` 条切片，而是围绕锚点窗口展开。
- 当 `selectedDate` 改变时，列表窗口会重算。
- 右键卡片会打开删除菜单。
- 删除非当前日期后，列表刷新但编辑区不切换。
- 删除当前日期后，编辑区进入空状态。

### 数据层

- `worklogApi.deleteEntry()` 的 mock 路径可删除条目。
- Rust `delete_entry` 命令能删除存在的文件，并允许不存在时安全返回。

### 不纳入本轮

- 不对虚拟列表性能做测试，因为本轮不做虚拟列表。
- 不对复杂菜单定位边界做像素级测试。
- 不对无限滚动做端到端性能测试，只验证范围扩展是否被触发。
